2. Problem Statement

The Revenue Integrity Challenge

Medicare Advantage plans rely on Hierarchical Condition Categories (HCCs) to receive risk-adjusted payments from CMS. Accurate HCC coding requires:

  • Identifying all documented conditions in clinical notes
  • Mapping to specific ICD-10-CM codes
  • Verifying adequate documentation (MEAT criteria)
  • Applying CMS hierarchy rules to determine payable categories
  • Calculating Risk Adjustment Factor (RAF) scores

Current Industry Pain Points

Missed HCCs = Lost Revenue

Studies show 15-30% of valid HCCs are missed during chart review. For a 100,000-member plan, this can mean $50M-$150M in unrealized risk adjustment payments annually.

Over-coding = Audit Risk

False positive HCC submissions trigger RADV audits with potential extrapolated penalties. Each unsupported HCC can cost $30K-$100K+ in audit recovery.

What MedInsight RA Engine Solves

  1. Automated HCC Discovery: ML model scans clinical notes to identify all candidate HCC categories
  2. Evidence-Based ICD Coding: Retrieval + LLM verification maps HCCs to specific ICD-10 codes with evidence spans
  3. Documentation Verification: MEAT criteria validation ensures coding defensibility
  4. Audit-Ready Output: Every prediction includes evidence spans, context flags, confidence scores
  5. HEDIS Gap Detection: Simultaneously identifies quality measure compliance gaps
  6. RAF Scoring: Instant computation of risk adjustment impact

4. Inference Pipeline

The RAEngine.analyze(note_text: str) -> dict function executes a 10-stage pipeline:

1
ML HCC Prediction
Bio_ClinicalBERT predicts top-20 HCC candidates with sigmoid scores. Section-aware tokenizer splits long notes into 8 overlapping 512-token chunks. ChunkAttentionPooler produces document-level representation.
2
ICD Retrieval
For each predicted HCC, retrieve top-20 candidate ICD-10 codes via CMS crosswalk + TF-IDF similarity. 7,903 ICD codes indexed. Section-aware boosting (Assessment 1.5x, Family History 0.7x).
3
Candidate Reranking
K=200 candidates narrowed to K=50 via embedding similarity, then to K=10 via section scoring and clinical context.
4
Span Proposal
SpanProposer identifies exact character-level evidence spans for each candidate ICD code within the clinical note text.
5
Context Gating (NegEx)
Deterministic ConText/NegEx tagger checks each evidence span for: negation (53 triggers), historical (23), hypothetical (26), family (18), uncertain (26). All-negated evidence = hard rejection.
6
LLM Verification
OpenAI gpt-4o verifies each ICD code with structured JSON output: evidence spans, MEAT elements, specificity choice, confidence score, reasoning. Results cached in SQLite.
7
ICD → HCC Mapping
Verified ICD codes mapped to HCC categories using official CMS V28 crosswalk (8,183 mappings).
8
V28 Hierarchy Suppression
Within 21 hierarchy groups, higher-severity HCC suppresses lower. E.g., HCC35 (Diabetes with major complications) suppresses HCC37 (Diabetes without complications).
9
RAF Scoring
Raw RAF = sum of all supported HCC coefficients. Final RAF = sum of payable HCC coefficients (post-hierarchy). Using official CMS community_nondual_aged model.
10
HEDIS Extraction
6 quality measures checked: CDC A1C, CDC Eye Exam, Controlling BP, Colorectal Screening, Breast Cancer Screening, Statin Therapy. Each classified as met/not_met/excluded/unknown.

Section-Aware Tokenizer

Recognizes 17 clinical section headers to intelligently split long notes at section boundaries:

Sections detected: Chief Complaint, HPI, Past Medical History, Medications, Allergies, Family History, Social History, Review of Systems, Physical Examination, Vitals, Labs, Imaging, Assessment, Plan, Discharge Diagnosis, Hospital Course, Procedures

Chunking strategy:

  • Chunk size: 512 tokens
  • Overlap: 64 tokens between chunks
  • Max chunks: 8 (caps at 4,096 effective tokens)
  • Prefers splitting at section boundaries
  • Each chunk wrapped with [CLS] and [SEP]

Key Bug Fixes That Enabled Improvement

Root Cause: Pipe-Delimited HCC Label Bug

The member_year_labels.csv used | as HCC separator (e.g., "HCC238|HCC37|HCC38"), but load_synthea_labels() only split by ;. This created:

  • 80 combo labels like "HCC298|HCC37" treated as single labels instead of individual codes
  • Model predicted index-based labels that never matched named HCC codes
  • Member-year evaluation showed 0 true positives

Fix: Added _split_hcc_codes() that normalizes both | and ; separators, producing correct individual HCC codes. Label mapping expanded from 80 combo labels to 115 individual V28 HCCs.

12. Context / NegEx Tagger

The ConText/NegEx tagger is a deterministic hard gate — if all evidence spans for an ICD code are negated, the code is marked supported=false regardless of LLM output.

Context Type# TriggersExample TriggersEffect
Negation53 pre + 12 post"no evidence of", "denies", "negative for", "without"Code not supported
Historical23"history of", "prior", "formerly", "resolved", "s/p"Code not current
Hypothetical26"rule out", "possible", "suspected", "consider"Code uncertain
Family18"family history of", "mother/father had", "familial"Not patient's condition
Uncertain26"uncertain", "equivocal", "probable", "likely"Insufficient certainty

Scope Rules

  • Pre-triggers: target must come after trigger, within 10-word window, same sentence
  • Post-triggers: target must come before trigger, same scoping
  • Scope terminators: "but", "however", "except", "yet" break trigger scope
  • Double-negation: "cannot rule out", "cannot exclude" → treat as uncertain
  • Section detection: "Family History" section auto-marks all spans as family=true

13. Evaluation Framework

Note-Level Evaluation

Command: python -m ra_engine.cli eval --dataset test.jsonl --output-dir eval_output

Runs full RA Engine analysis on labeled JSONL dataset and computes:

  • Per-example TP/FP/FN counts
  • RAF impact per prediction (TP_raf, FP_raf, FN_raf)
  • Macro & micro precision, recall, F1
  • Per-HCC confusion matrix
  • Top missed HCCs sorted by RAF impact
  • Top false positives sorted by audit risk

Output files: errors_by_raf.csv, fp_by_audit_risk.csv, confusion_by_hcc.json, llm_stats.json

Member-Year Evaluation

Command: python -m ra_engine.cli eval-member-year --dataset member_year_dir/ --output-dir eval_output

Aggregates HCC predictions across all encounters for each member within a year:

  • HCC Union: Payable HCCs from all encounters merged (deduped)
  • HEDIS "Met Wins": If any encounter shows measure met, member-year is met
  • RAF-Weighted Metrics: Recall/precision weighted by CMS coefficient
  • Ground Truth: From labels.csv with ground_truth_hccs column

Required files: member_year.jsonl (encounters per member-year) + labels.csv (ground truth)

Batch Processing Pattern

import json
from ra_engine.inference.analyze import RAEngine

engine = RAEngine(
    model_dir="models/ra_engine/hcc_predictor_mixed_large/best",
    reference_dir="data/reference"
)

# Process member encounters from MedInsight platform
for encounter in medinsight_encounters:
    result = engine.analyze(encounter["note_text"])

    # Extract for platform JSON
    platform_record = {
        "encounter_id": encounter["id"],
        "member_id": encounter["member_id"],
        "supported_icds": [
            {"code": icd["code"], "confidence": icd["confidence"]}
            for icd in result["icd_layer"]["supported_icd10"]
            if icd["supported"]
        ],
        "payable_hccs": [
            hcc["hcc"] for hcc in result["hcc_layer"]["payable_hccs_final"]
        ],
        "raf_score": result["raf_layer"]["raf_payable_final"],
        "hedis_gaps": [
            {"measure": m["measure_id"], "status": m["status"]}
            for m in result["hedis_layer"]["measures"]
            if m["status"] == "not_met"
        ],
        "audit_score": result["audit_pack"]["evidence_validity_rate"]
    }

    # Send to MedInsight platform
    platform.submit(platform_record)
