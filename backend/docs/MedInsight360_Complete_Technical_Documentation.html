Platform Overview

MedInsight 360 is a production-grade, AI-powered medical chart analysis platform that combines Risk Adjustment (HCC coding) and Quality Intelligence (HEDIS measures) into a single unified system. It ingests medical chart PDFs, applies multi-layer AI extraction, ML prediction, deterministic verification, and produces actionable Payable HCC Packs and HEDIS Gap reports.

Key Capabilities

HCC Risk Adjustment

Ensemble HCC prediction combining LLM extraction, BioClinicalBERT ML, and TF-IDF retrieval. V28 ICD→HCC mapping with hierarchy suppression. Full RAF score computation with MEAT evidence validation.

HEDIS Quality Measures

Automated evaluation of 100+ quality measures. Gap identification with actionable recommendations. Evidence-backed compliance tracking with PDF source references.

PDF Intelligence

Per-page quality scoring, GPT-4o Vision OCR fallback, MiniMax single-pass extraction of all clinical facts with exact quote preservation and character offset tracking.

Coder Review Workflow

Full accept/reject/edit workflow for diagnoses, HCCs, and HEDIS measures. Audit trail logging every action. Decision trace explaining why each coding decision was made.

Interactive PDF Viewer

Split-pane viewer with evidence highlighting. Click any diagnosis, HCC, or HEDIS item to jump to the supporting text in the PDF with glow animation.

Configurable Everything

Runtime feature flags, LLM provider switching (OpenAI/Azure/Gemini), editable prompts, adjustable thresholds — all via dashboard without code changes.

HCC Ensemble: The Dual Approach

This is the competitive advantage. Rather than relying solely on LLM-extracted ICD codes (which can hallucinate), MedInsight 360 uses a 3-source ensemble:

Source 1: LLM Extraction

The MiniMax extractor reads clinical text and suggests ICD-10 codes directly. Good for obvious diagnoses but can miss subtle ones or hallucinate specificity.

Source 2: BioClinicalBERT ML

Multi-label classifier predicts HCC categories (not ICDs) from raw text. Catches patterns LLMs miss in long documents. 115 payable HCC categories.

Source 3: TF-IDF Retrieval

For each ML-predicted HCC, retrieve the most similar ICD-10 codes from the 7,903-code catalog by matching clinical language to ICD descriptions. Gets the right code specificity.

Candidates from all 3 sources are merged, deduplicated, filtered through NegEx gating, then verified by GPT-4o with MEAT evidence checks.

Negation: The 6-Status Model

MedInsight 360 does not simplify to binary (present/absent). Every clinical mention gets one of 6 polarity statuses:

StatusExamplePayable?
active"Patient has type 2 diabetes with chronic complications"Yes
negated"No evidence of CHF", "Ruled out PE"No
resolved"Diabetes resolved after pancreas transplant"No
historical"History of stroke in 2018"Maybe*
family_history"Mother had breast cancer"No
uncertain"Possible CHF", "R/O pulmonary embolism"No

*Historical may be payable if recaptured with current-year evidence showing ongoing management.

MEAT Validation

Every payable ICD must have at least one MEAT element documented:

LetterElementEvidence Examples
MMonitoredLabs ordered, vital signs tracked, follow-up scheduled
EEvaluatedSymptoms assessed, exam findings documented
AAssessedDiagnosis in assessment/plan, clinical reasoning documented
TTreatedMedications prescribed, procedures performed, therapy ordered

Service Layer

The service layer sits between API routers and repositories, containing all business logic.

ChartService

File: services/chart_service.py

Purpose: Chart lifecycle management — create, list, get, delete.

Key Methods

  • create_chart(filename, file_path, ...) — Creates chart record + initial pipeline run
  • list_charts(offset, limit, status) — Paginated list enriched with RAF + HEDIS summaries + patient data
  • get_chart_summary(chart_id) — Full chart details with pages, runs, patient
  • delete_chart(chart_id) — Cascading delete of all related data (30+ table cleanup)
The list_charts method performs eager loading of the Patient relationship via selectinload to avoid async lazy-loading errors, and enriches each chart with RAF summary and HEDIS summary data from their respective repositories.

AssertionService

File: services/assertion_service.py

Purpose: Query and serialize clinical data (assertions) for API responses.

Key Methods

  • get_diagnoses(chart_id) — Filter assertions by diagnosis category, serialize with ICD codes, evidence
  • get_medications(chart_id) — Medication assertions with dose/instructions
  • get_vitals(chart_id) — Vital signs with BP parsing, BMI calculation
  • get_labs(chart_id) — Lab results with values, ranges, HEDIS measure linking
  • get_encounters(chart_id) — Encounter timeline derived from assertion dates

RiskService

File: services/risk_service.py

Purpose: Assemble HCC Pack, RAF summary, and hierarchy details.

Key Methods

  • get_hcc_pack(chart_id) — Complete payable HCC pack with supported ICDs, MEAT evidence, suppressed list, RAF summary
  • get_raf_summary(chart_id) — RAF breakdown (total, demographic, HCC, per-HCC weights)
  • get_hierarchy(chart_id) — Which HCCs suppressed which, and why

HEDISService

File: services/hedis_service.py

Purpose: HEDIS measure results, gap detection, evidence formatting.

Key Methods

  • get_measures(chart_id) — All measures with met/gap/excluded status
  • get_gaps(chart_id) — Identified care gaps with action items
  • get_evidence(chart_id) — Met measures with supporting evidence
  • get_summary(chart_id) — Aggregate counts

ReviewService

File: services/review_service.py

Purpose: Full coder review CRUD for diagnoses, HCCs, and HEDIS measures.

Key Methods

  • add_diagnosis(chart_id, icd10_code, ...) — Coder adds new diagnosis
  • accept_diagnosis(assertion_id, reviewer, notes) — Mark as accepted
  • reject_diagnosis(assertion_id, reviewer, reason) — Mark as rejected
  • accept_hcc(hcc_id, reviewer, notes) — Accept HCC code
  • reject_hcc(hcc_id, reviewer, reason) — Reject HCC code
  • save_document(chart_id, payload) — Bulk save entire document review
Every review action creates an audit log entry and a review_action record for full traceability.

Functionality Guide

What the system does, how it gets data, what users see.

1. Chart Upload & Processing

What Happens

  1. User drags a medical chart PDF into the upload zone (or clicks to browse)
  2. Frontend POSTs the file as multipart/form-data to /api/charts/upload
  3. Backend saves the file to uploads/, creates a charts record and a pipeline_runs record
  4. Background task launches the Orchestrator pipeline (async)
  5. Frontend polls /api/charts/{id} every 5 seconds until status = "completed"
  6. Chart card updates in real-time: status badge changes, RAF/HEDIS badges appear

Data Flow

User → [PDF file] → POST /api/charts/upload
                           │
                           ▼
                    charts table (status: uploaded)
                    pipeline_runs table (status: running)
                           │
                    Orchestrator.process_chart()
                           │
                    ┌──────┴───────┐
                    │ Step 1-6     │  (see Pipeline Flow tab)
                    └──────┬───────┘
                           │
                    charts table (status: completed)
                    assertions table (500+ rows per chart)
                    payable_hccs, raf_summaries, hedis_results tables

2. Chart List & Search

What Users See

A grid of compact chart cards showing: filename, patient name, status badge, relative time, RAF score, HCC count, HEDIS MET/GAP counts, page count.

How Search Works

  • Frontend fetches all charts from GET /api/charts
  • Backend returns charts enriched with patient_name, raf_summary, hedis_summary
  • Frontend's searchItems() filters client-side by matching against: chart_id, filename, patient_name, status
  • Advanced filters: HCC code, diagnosis text, high risk (RAF ≥ 2.0), has HEDIS gaps

How Data Gets There

Backend ChartService.list_charts() does:

  1. Query charts with selectinload(Chart.patient) for eager loading
  2. For each chart, fetch raf_summaries from RAFRepository
  3. For each chart, fetch hedis_summaries from HEDISRepository
  4. Serialize with patient_name, patient_dob, raf_summary, hedis_summary

3. PDF Viewer with Evidence Highlighting

What Users See

Split-pane view: PDF on the left, analysis panels on the right. Users can:

  • Navigate pages, zoom in/out, fit to width/page
  • Search text in PDF with match highlighting (yellow glow)
  • See evidence highlights when clicking diagnoses or HCC codes
  • Select text in PDF to add a new diagnosis
  • Hover highlights to see evidence popovers

How Highlighting Works

  1. User clicks "View Evidence" on a diagnosis or HCC code
  2. Component calls pdfStore.navigateToText(quote, type, label, meta)
  3. PDFViewer receives pendingEvidenceSearch from store
  4. Searches all PDF pages for the exact quote text
  5. Uses DOM text layer for pixel-perfect positioning
  6. Creates a PDFHighlight overlay with glow animation
  7. Scrolls to the matching page and centers the highlight

How PDF Search Works

  1. User types in search box (debounced 300ms)
  2. performSearch() iterates all PDF pages via pdfDocProxy.getPage()
  3. Extracts text content, finds all occurrences of the query
  4. Creates search highlights (yellow) and stores matches
  5. Results dropdown shows matches grouped by page
  6. Clicking a result navigates to that page and activates the highlight

4. HCC Pack Analysis

What Users See

  • RAF Summary Card: Total RAF score (1.847), demographic RAF, HCC RAF, payable/suppressed counts
  • Payable HCC Accordion: Each HCC with code, description, RAF weight, audit risk level
  • Supporting ICDs: Per HCC, list of ICD-10 codes with confidence (ML + LLM), MEAT evidence, evidence quotes with page references
  • Review Actions: Accept/Reject buttons per HCC with comments

How Data Gets There

Frontend: useHCCPack(chartId)
  → GET /api/risk/{chartId}/hcc-pack
    → RiskService.get_hcc_pack()
      → HCCRepository.get_payable(chart_id) → payable_hccs rows
      → HCCRepository.get_suppressed(chart_id) → suppressed_hccs rows
      → RAFRepository.get_summary(chart_id) → raf_summaries row
      → RAFRepository.get_hierarchy_log(chart_id) → hierarchy entries
    → Assemble into HCC Pack JSON structure

5. HEDIS Quality Measures

What Users See

  • Summary Cards: Total measures, Met count (green), Gap count (red), Excluded count
  • Measure Cards: Each measure with status badge, eligibility, evidence items, missing evidence
  • Evidence Snippets: Date, value, source section, page reference, exact quote
  • Gap Recommendations: What's missing, what action is needed

How Data Gets There

Frontend: useHEDISMeasures(chartId)
  → GET /api/hedis/{chartId}/measures
  → GET /api/hedis/{chartId}/gaps
    → HEDISService.get_measures()
      → HEDISRepository.get_by_chart(chart_id) → hedis_results rows
      → HEDISRepository.get_summary(chart_id) → hedis_summaries row
    → Format measures with status, evidence, gaps

6. Coder Review Workflow

What Users Can Do

  • Accept a diagnosis/HCC/HEDIS measure (confirm AI extraction is correct)
  • Reject with reason (AI made an error)
  • Edit — modify ICD code, description, evidence
  • Add New — manually add a diagnosis the AI missed
  • Bulk Save — save all review decisions for the entire document

How It Works

User clicks "Accept" on diagnosis
  → ReviewModal opens with optional notes
  → User confirms
  → PUT /api/review/diagnosis/{id} { status: "accepted", reviewer, notes }
    → ReviewService.accept_diagnosis()
      → Update assertion.review_status = "accepted"
      → Create review_actions record
      → Create audit_log entry
  → React Query invalidates diagnosis cache
  → UI updates with "Accepted" badge

cd medinsight360_codex
