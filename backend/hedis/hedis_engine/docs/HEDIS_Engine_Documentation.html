Project Objectives

Mission Statement

Build a production-grade, deterministic HEDIS measure evaluation engine that assesses member eligibility, compliance, and quality gaps from extracted clinical evidence. The system is fully rule-based (no ML), audit-friendly, and designed to scale from 35 to 50+ measures without architectural changes.

Core Requirements Delivered

Done Measure Evaluation Engine

For each member and measure, the engine answers:

  • Is this measure applicable (eligible)?
  • Is the member compliant? If not, what gap remains?
  • What evidence supports compliance (with dates + source refs)?
  • What evidence supports denominator inclusion/exclusion?
  • What data is missing that prevents determination?
Done Data-Driven DSL

Measures are defined in YAML files — no code changes needed to add, modify, or remove measures. Value sets are JSON files that can be swapped with official NCQA-licensed sets.

Done Deterministic & Auditable

Zero ML for eligibility/compliance decisions. Every decision produces a structured trace describing which rule fired, which evidence was used, and which dates were applied.

Done 35 Measures Across 8 Domains

Diabetes, hypertension, preventive screenings, childhood/adolescent/adult immunizations, depression, medication management, well-visits, osteoporosis, tobacco, and care transitions.

Done Evidence Citation

Every matched code/value includes PDF file, page number, exact quote, character offsets, code system, and event date for full audit traceability.

Done Flexible Upstream Integration

Adapter converts heterogeneous upstream JSON ("assertions") into normalized event stores. Handles 5 date formats, auto-classifies medications, labs, vitals, and vaccines.

NCQA Licensing Compliance

Official HEDIS specifications and value set files are NCQA-licensed. This implementation:

  • Implements the engine architecture, rule DSL, and measure catalog scaffolding
  • Populates measures using publicly available descriptions (CMS quality measures, CDC ACIP schedules, USPSTF recommendations)
  • Designs the code so official value sets/specs can be dropped in later without refactor
  • Does not copy licensed NCQA spec text verbatim

Evaluation Pipeline

Every measure follows a 4-step deterministic evaluation flow. If any step produces a terminal result, subsequent steps are skipped.

Step 1
Eligibility
Age + Gender gate
→
Step 2
Denominator
Condition / event required
→
Step 3
Exclusions
Hospice, pregnancy, etc.
→
Step 4
Numerator
Compliance check
Step 1: Eligibility
  • age_between(min, max, as_of_date) — Check member age at year-end (or custom date)
  • gender_in(allowed_set) — Verify gender matches measure requirement
  • continuous_enrollment() — Stub (assumes enrolled); production would check enrollment feed

Terminal: If age/gender fail → NOT APPLICABLE

Step 2: Denominator
  • Age-only: Population-based measures (e.g., COL, BCS) — all eligible members
  • Diagnosis required: Must have specific ICD codes (e.g., diabetes for CDC measures)
  • Procedure required: Must have had specific procedures
  • Encounter required: Must have visit type (e.g., outpatient for BMI)

Terminal: If denominator not met → NOT APPLICABLE

Step 3: Exclusions
  • Check each exclusion criterion in sequence
  • Common: hospice care, pregnancy, ESRD, prior surgery
  • First match triggers early exit

Terminal: If any exclusion matched → EXCLUDED

Step 4: Numerator
  • all_of: Every criterion must be met (conjunction)
  • any_of: At least one criterion met (disjunction)
  • For each unmet criterion, a GapDetail is created
  • Evidence is collected for every matched criterion

All met → MET   Any missing → GAP   No criteria defined → INDETERMINATE

Architecture

Value sets decouple code-level specifics from measure logic. Each measure YAML references abstract IDs like VS_DIABETES_ICD10. The engine loads codes from JSON files at runtime. Official NCQA-licensed value sets can replace placeholders without any code changes.

Code Normalization

All codes are normalized before comparison: strip() → upper() → remove dots. So e11.9, E11.9, and E119 all match.

Loader Behavior

  • Value sets are cached in memory after first load
  • Missing value set files return empty set (engine doesn't crash)
  • Loader searches by exact filename, then scans all JSON files for matching "id" field

Key Design Details

Medication Class Aliases

The medication_present primitive supports class aliases for compound categories:

_CLASS_ALIASES = {
  "ace_arb": {"ace_inhibitor", "arb", "ace_arb"},
  "antihypertensive": {"ace_inhibitor", "arb", "beta_blocker", "ccb", "diuretic"},
  "diabetes_med": {"metformin", "insulin", "sulfonylurea", "sglt2", "glp1"},
}

This allows medication_class: "ace_arb" in YAML to match both ACE inhibitors (lisinopril) and ARBs (losartan).

Calendar-Accurate Time Windows

The lookback_window() function uses proper calendar month subtraction, not days * 30:

lookback_window(end_date=2025-12-31, months=12)
  → Start: 2024-12-31 (calendar-accurate)

lookback_from_year_end(year=2025, months=3)
  → Window: 2025-09-30 .. 2025-12-31

extended_window(year=2025, lookback_years=10)
  → Window: 2015-01-01 .. 2025-12-31

Programmatic API

from hedis_engine.adapters.assertion_adapter import load_assertions_file
from hedis_engine.engine import HedisEngine

store = load_assertions_file("member.json")
engine = HedisEngine(measurement_year=2025)
results = engine.evaluate_member(store)

for m in results.measures:
    if m.status.value == "gap":
        print(f"{m.measure_id}: {[g.description for g in m.gaps]}")

Extensibility Guide

Adding a New Measure

Step 1: Create YAML Definition
# catalog/MY_NEW_MEASURE.yaml
id: "MY-MEASURE"
name: "My Clinical Measure"
version: "2025"
domain: "my_domain"
description: "Description based on public clinical guidelines"

eligibility:
  age: {min: 18, max: 75, as_of: "12-31"}
  gender: ["male", "female"]

denominator:
  requires_diagnosis:
    valueset: "VS_MY_CONDITION"
    lookback_months: 12

exclusions:
  - type: "diagnosis"
    valueset: "VS_HOSPICE"
    description: "Hospice care"

numerator:
  any_of:
    - type: "procedure"
      valueset: "VS_MY_PROCEDURE"
      window_months: 12
    - type: "lab_value"
      lab: "MY_LAB"
      comparator: "lt"
      threshold: 5.0
      window_months: 12

valuesets_needed:
  - VS_MY_CONDITION
  - VS_MY_PROCEDURE
  - VS_HOSPICE
Step 2: Create Value Set (if needed)
// valuesets/VS_MY_CONDITION.json
{
  "id": "VS_MY_CONDITION",
  "name": "My Condition Codes",
  "code_system": "icd10cm",
  "placeholder": true,
  "codes": [
    {"code": "K21.0", "description": "GERD with esophagitis"}
  ]
}
Step 3: No Code Changes Required

The engine auto-loads all YAML files from the catalog directory on initialization. Run the CLI and the new measure will be evaluated automatically.

Adding a New Rule Primitive

  1. Add function to primitives.py returning (bool, str, list[EvidenceUsed])
  2. Add corresponding criterion type to measure_def.py parser
  3. Add dispatch case in engine.py _eval_criterion()
  4. Add test in test_primitives.py

Replacing Placeholder Value Sets

Replace any valuesets/VS_*.json file with official NCQA-licensed codes using the same JSON structure. The loader reads the "codes" array and normalizes codes automatically.
