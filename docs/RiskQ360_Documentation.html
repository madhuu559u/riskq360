What is RiskQ360?

RiskQ360 is a unified clinical documentation analytics platform that processes medical chart PDFs through an AI-powered pipeline to produce:

  • Payable HCC Packs -- CMS-HCC V28 risk adjustment with ICD-10 mapping, hierarchy suppression, and RAF score computation
  • HEDIS Quality Measures -- Automated measure evaluation for CBP, CDC, COL, BCS, CCS, and more
  • Clinical Assertions -- Structured extraction of diagnoses, medications, vitals, labs, and encounters with full provenance
  • Audit-Ready Evidence -- MEAT validation, evidence spans, negation detection, and review workflows

Key Concepts

Assertion-Centric Architecture

The assertions table is the central entity in RiskQ360. Every clinical fact extracted from a medical chart (diagnosis, medication, vital sign, lab result, etc.) is stored as a single assertion row with full provenance including exact quotes, page numbers, character offsets, and enrichment data (ICD codes, dates, negation status).

MiniMax Single-Pass Extraction

Rather than running separate LLM pipelines for different clinical categories, RiskQ360 uses a unified single-pass extraction model that extracts ALL assertion types in one LLM call per text chunk. This approach reduces API costs, improves consistency, and enables cross-reference between clinical facts.

6-Status Negation Model

Every clinical assertion is tagged with one of six polarity statuses: active negated resolved historical family_history uncertain. Only active assertions are candidates for payable HCCs. Historical assertions may qualify if supported by current-year encounter evidence.

API Client Layer

A centralized Axios instance at src/api/client.ts handles all HTTP communication with the FastAPI backend. It includes:

  • Base URL from environment variable VITE_API_BASE_URL (defaults to /api)
  • 60-second request timeout
  • Automatic error interception with Mantine notification toasts
  • HTTP status-specific error titles (404, 422, 500, network errors)

Individual API modules (charts.ts, clinical.ts, risk.ts, etc.) import the shared client and expose typed functions for each endpoint.

Layered Architecture

Request Flow

1

Router Layer

FastAPI route handlers validate request parameters, delegate to service/repository layer, and return Pydantic response models.

2

Service / Core Layer

Business logic in the orchestrator, HCC bridge, HEDIS bridge, and ensemble engine. Coordinates between extraction, ML, and decisioning.

3

Repository Layer

Data access objects that encapsulate all SQLAlchemy queries. Each repository handles one domain entity (charts, assertions, HCCs, etc.).

4

Database Layer

SQLAlchemy ORM models, connection pooling (asyncpg/psycopg2), session management, and Alembic migrations.

Configuration System

Settings are managed via Pydantic BaseSettings with hierarchical configuration. See the Configuration tab for full details. The Settings root class aggregates:

  • DatabaseSettings -- PostgreSQL connection parameters
  • LLMSettings -- LLM provider selection and API keys
  • MLSettings -- ML model paths and thresholds
  • FeatureFlags -- Runtime-toggleable feature flags
  • PipelineSettings -- Chunk size, concurrency, thresholds
  • ServerSettings -- Host, port, reload
  • PathSettings -- Output, log, upload directories

System Architecture

End-to-end data flow from PDF upload through LLM extraction, ML prediction, and decisioning to payable HCC/HEDIS output.

Pipeline Data Flow

PDF Upload | v [1. INGESTION] --------+-------- PDF Processor | | Quality Scorer (0-100 per page) | | OCR Engine (GPT-4o Vision fallback) | +-------- Text Normalizer v [2. EXTRACTION] -------+-------- MiniMax Single-Pass LLM | | One call per text chunk | | Extracts ALL assertion types: | | - Diagnoses + ICD codes | | - Medications (normalized) | | - Vitals (structured BP, BMI) | | - Lab results (values, units) | | - Encounters (DOS, provider) | +-------- Assertions + provenance (quotes, pages, offsets) v [3. ENRICHMENT] ------+-------- Deterministic post-processing | | Quote repair + matching | | Date inference | | Negation detection (6-status) | | Condition grouping | +-------- Code assignment (ICD, CPT, HCPCS) v [4. HCC BRIDGE] ------+-------- ICD-to-HCC V28 Mapping | | Hierarchy Suppression | | RAF Score Computation | | Ensemble Scoring (LLM + TF-IDF + BERT) | +-------- Payable HCC Pack generation v [5. HEDIS BRIDGE] ----+-------- Measure Eligibility Check | | Evidence Matching (valuesets) | | Gap Identification | +-------- Quality Pack generation v [6. PERSISTENCE] ----+-------- PostgreSQL (30+ tables) | | Assertions (central) | | Payable HCCs + RAF Summaries | | HEDIS Results + Summaries | +-------- Audit Logs + Pipeline Logs v API / Dashboard Output

Assertion-Centric Architecture

The assertions table is the single source of truth for all clinical facts. Every pipeline stage reads from and writes back to this table. This design provides:

  • Single audit trail -- Every clinical fact has provenance (exact_quote, page_number, char_start/end)
  • Cross-reference -- Diagnoses, medications, and labs can be linked via condition_group_id
  • Review workflow -- Every assertion has review_status (pending/approved/rejected)
  • Flexible querying -- Category-based filtering for diagnoses, vitals, labs, medications, etc.
+---------------------------+ | ASSERTIONS | | (Central Table) | +---------------------------+ | +------------------+------------------+ | | | v v v +------------+ +------------+ +------------+ | Payable | | Condition | | HEDIS | | HCCs | | Groups | | Results | +------------+ +------------+ +------------+ | | v v +------------+ +------------+ | RAF | | HEDIS | | Summaries | | Summaries | +------------+ +------------+

MiniMax Single-Pass Extraction

The MiniMax architecture extracts ALL clinical assertion types in a single LLM call per text chunk, rather than running separate specialized pipelines. This provides:

  • Cost efficiency -- One API call vs. five separate pipeline calls per chunk
  • Consistency -- Cross-category references are preserved within the same context window
  • Speed -- Parallel chunk processing without pipeline coordination overhead
  • Structured output -- JSON schema enforcement ensures consistent assertion shape

Extraction Categories

Each assertion is classified into one of these categories:

diagnosis medication vital_sign lab_result procedure encounter allergy social_history family_history screening immunization

HCC Ensemble Strategy

RiskQ360 uses a multi-source ensemble for HCC identification, combining three evidence channels:

1

LLM Direct Extraction

MiniMax extraction outputs ICD-10 codes directly from clinical text. Good for obvious diagnoses but can hallucinate codes.

2

TF-IDF ICD Retrieval

Matches clinical language to ICD-10 descriptions using TF-IDF similarity over the 7,903-code catalog. Finds correct code specificity.

3

BioClinicalBERT HCC Prediction

Multi-label classifier predicts HCC categories directly from raw text. Catches patterns LLMs miss in long documents.

The ensemble merges candidates from all three sources, deduplicates, applies NegEx gating, and produces confidence-weighted final HCCs.

LLM Configuration

Provider Selection

Set ACTIVE_LLM_PROVIDER to one of:

  • openai -- Direct OpenAI API (requires OPENAI_API_KEY)
  • azure -- Azure OpenAI Service (requires AZURE_OPENAI_ENDPOINT, AZURE_OPENAI_KEY)
  • gemini -- Google Gemini (requires GOOGLE_GEMINI_KEY)

The provider can be changed at runtime via the Configuration API or dashboard without restarting the server.

HEDIS Configuration

Measurement Year

Set via MEASUREMENT_YEAR environment variable (default: 2026). This controls which HEDIS measure definitions are active and what date ranges are evaluated.

Measure Definitions

HEDIS measures are stored in the hedis_measure_definitions table and loaded from YAML files at startup. Measures can be enabled/disabled per profile via the API.

Value Sets

Code value sets for each measure are stored in hedis_valuesets with code_system and payload_json containing the actual codes (ICD-10, CPT, LOINC, etc.).
